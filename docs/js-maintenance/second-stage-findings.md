# 第二阶段：官方运行环境与修改版差异结论

## 1. 这轮解决了什么

本阶段把修改版 `game.js 1.0.2` 与当前官方 H5 运行环境分开观察，补充了三类证据：

1. 当前官方页面真实加载的公共脚本、大小和 SHA-256；
2. 无登录、无购买、只读浏览器环境中的真实资源和接口活动；
3. 修改版与当前官方版使用同一静态重组规则后的结构差异。

完整第三方源码没有复制进知识库。浏览器报告只保留 URL 结构、参数名和经过脱敏的参数值。

## 2. 当前官方页面资源图

当前页面加载的主要脚本包括：

```text
官方 game.js
SALInterface.js
webGLLib.js
Common.js
GameInfoData.js
InfoTool.js
OrgWeb.js
jquery
spine-core
utfx
ssologin
统计 SDK
```

维护时应当把它们分成三层：

| 层 | 代表文件 | 主要风险 |
|---|---|---|
| 页面启动与登录 | `Common.js`、`InfoTool.js`、`OrgWeb.js`、`ssologin.js` | 页面入口、登录、全局变量和加载顺序变化 |
| 平台与渲染 | `SALInterface.js`、`webGLLib.js` | SAL 参数、Canvas/WebGL、音视频和资源管理变化 |
| 引擎与作品 | `game.js`、作品二进制及共享资源 | 用户、商城、存档、场景和业务流程变化 |

因此以后不能只监控 `game.js`。出现白屏或功能失效时，应先根据变更层确定排查范围。

## 3. 三个 Webpack 依赖的最终定性

修改版和当前官方版具有相同结构：

```text
chunk 全局：webpackChunk_lodash_modules
chunk 名称：ondemand.s
入口模块：71269
依赖顺序：36728 → 6886 → 75640
```

三项依赖在两份 `game.js` 中都没有定义，调用返回值也都被丢弃。

这说明它们应归类为：

> 入口阶段的外部副作用依赖。

它们更可能承担 Polyfill、运行环境初始化、完整性准备或调试环境准备，而不是向商城、用户或存档代码返回业务对象。

维护规则：

- 不把编号与具体业务功能绑定；
- 只检查依赖顺序是否仍存在；
- 检查页面公共环境是否在入口模块执行前准备完成；
- 编号变化时重新做入口结构分析，不直接替换编号。

浏览器中的 chunk 队列没有长期暴露可调用的 `require.m` 注册表，所以当前无法仅靠页面运行时读取三个模块的完整工厂源码。这不影响将它们归类为外部副作用依赖。

## 4. SAL 平台能力结论

| 指标 | 修改版 | 当前官方 |
|---|---:|---:|
| 直接识别 SAL | 122 | 148 |
| 简单字符串重组后 SAL | 150 | 150 |
| 修改版独有 SAL | 0 | - |
| 当前官方独有 SAL | - | 0 |

结论：两份引擎的平台接口能力面一致，均覆盖 150 个 SAL 标识符。

后续更新时，SAL 的判断优先级是：

1. 接口集合是否变化；
2. 同名接口参数是否变化；
3. 返回值和回调时机是否变化；
4. `SALInterface.js` 实现是否变化。

不能因为接口名仍存在，就认定兼容性一定正常。

## 5. 当前最值得关注的结构差异

两份文件不是同一官方构建日期，以下只能作为维护信号，不能直接认定为自定义修改。

### 用户与身份

修改版相对当前官方版：

- `userData` 多 60 次；
- `uid` 多 4 次；
- `guid` 多 5 次；
- `isLogin` 少 7 次。

这说明用户对象访问和状态适配是高优先级版本敏感区域。

### 鲜花与数值

修改版中明显更多的字段：

```text
totalFlower
wildFlower
tempFlower
realFlower
freshFlower
```

其中部分字段在当前官方版中为零或显著更少。后续失效时，应优先检查用户数据快照、访问器和数值来源，不要只修改菜单显示。

### 商城与订单

两版都大量使用：

```text
mallViewData
itemPrice
goods
goods_id
createBuyOrder
get_goods_list
```

出现次数差异不大，说明商城主体结构仍相近，但订单动作和响应分支存在版本差异。维护时应比较对象结构和响应字段，不应只比较接口路径。

### 存档与设置

修改版中的 `saveData`、`SaveData`、`showLocal` 使用更多。后续出现存档列表、自动存档或本地显示异常时，应同时排查：

- SAL Storage；
- 本地 Storage；
- 存档索引；
- 显示状态字段；
- 云端与本地分支。

### 渲染循环

`requestAnimationFrame` 的静态出现次数差距很大：修改版 1048，当前官方 331。

这个数字可能受混淆和版本生成方式影响，不能直接解释为运行次数增加，但它提示渲染/调度实现发生过明显重组。若未来出现卡顿、黑屏、后台恢复异常或耗电上升，应把帧循环作为独立故障层。

## 6. 真实浏览器观测结论

只读浏览器探针确认：

- 官方页面能够进入引擎加载阶段；
- 官方 `game.js`、SAL、WebGL 库和作品资源均被请求；
- 后续会出现用户状态、商城状态、存档、配置、统计和作品资源请求；
- 页面存在调试环境检测；
- 报告不会保存 token、签名、nonce、运行密钥等查询值。

浏览器探针的作用是确认“实际运行时发生了什么”，不是模拟登录、购买或修改返回值。

## 7. 后续更新的最短排查路径

```text
页面 HTML/脚本清单变化？
├─ 是 → 检查入口、公共脚本和加载顺序
└─ 否
   ↓
SALInterface/webGLLib 变化？
├─ 是 → 复测平台桥接、渲染、媒体和存储
└─ 否
   ↓
官方 game.js 变化？
├─ 是 → 生成 SAL、字段、路径和入口结构差异
└─ 否
   ↓
作品二进制或接口响应变化？
├─ 是 → 检查作品版本和数据结构
└─ 否
   ↓
检查 noname.js 注入时机、对象访问器和本地状态
```

## 8. 仍缺少的关键证据

1. 修改版 `1.0.2` 对应日期的官方原始 `game.js`；
2. 三个外部副作用依赖所在的历史基础 bundle；
3. 各主要功能在真实安卓 WebView 中的调用时序；
4. 多个不同作品和播放器版本的差异样本；
5. RC4/数组索引运行时解码出的剩余字符串。

当前知识库已经能支持版本变化定位、故障分层和更新前风险判断，但不会在缺少同版本官方基线时自动生成修改补丁。
